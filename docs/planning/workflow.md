# Workflow

This workflow is intended to keep contributions focused, traceable, and easy to review.

## Working an Issue

1. Confirm the scope of the issue and any linked documentation.
2. Create or switch to a branch named after the issue.
3. Make focused changes that solve the issue without bundling unrelated work.
4. Update docs when the change affects design, planning, usage, or workflow.
5. Run relevant checks or manual validation for the scope of the change.
6. Open a pull request and link the issue using `Closes #<number>` where appropriate.

## Pull Request Expectations

- Keep the PR focused on one logical outcome.
- Summarise what changed and why.
- Note how the change was validated.
- Flag any assumptions, limitations, or follow-up work.

## Definition of Done

An issue is considered done when:

- the scoped change is complete
- the repository still reads cleanly and consistently
- relevant documentation has been updated
- validation has been performed at an appropriate level for the task
- the pull request clearly links the work back to the issue
